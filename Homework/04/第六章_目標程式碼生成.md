## 6.1 Introduction to Code Generation

Code generation is the final phase of compilation, transforming the optimized intermediate representation into executable machine code for a specific target architecture.

```
LLVM IR → Optimization Passes → Selection → Register Allocation → Assembly/Object Code
```

### 6.1.1 Code Generator Responsibilities

| Responsibility | Description |
|---------------|-------------|
| **Instruction Selection** | Map IR instructions to target instructions |
| **Register Allocation** | Assign virtual registers to physical registers or stack |
| **Instruction Scheduling** | Order instructions for pipeline efficiency |
| **Calling Conventions** | Handle parameter passing and return values |
| **Stack Frame Layout** | Manage local variables and saved registers |

### 6.1.2 Target Independence vs. Dependence

```
                    ┌─────────────────────┐
                    │   LLVM IR (High)    │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   Optimization      │
                    │   (Target-Indep)    │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
    ┌─────────▼─────────┐ ┌────▼────┐ ┌────────▼────────┐
    │    x86/x86-64    │ │   ARM   │ │     RISC-V       │
    │    (Selection)   │ │         │ │                  │
    └─────────┬─────────┘ └────┬────┘ └────────┬────────┘
              │                │                │
    ┌─────────▼─────────┐ ┌────▼────┐ ┌────────▼────────┐
    │  Register Alloc   │ │         │ │                  │
    └─────────┬─────────┘ └────┬────┘ └────────┬────────┘
              │                │                │
    ┌─────────▼─────────┐ ┌────▼────┐ ┌────────▼────────┐
    │   Assembly/Obj    │ │         │ │                  │
    └───────────────────┘ └─────────┘ └──────────────────┘
```

## 6.2 LLVM Target Description

LLVM provides a declarative target description system that allows adding new architectures without writing C++ code.

### 6.2.1 Target Registration

```cpp
// Get target machine for host
Triple targetTriple = Triple::getDefaultTriple();
std::string error;
const Target* target = TargetRegistry::lookupTarget(targetTriple.getArch(), error);

if (!target) {
    errs() << "Error: " << error << "\n";
    return nullptr;
}

// Create target machine
TargetMachine* tm = target->createTargetMachine(
    targetTriple,           // Triple (e.g., x86_64-pc-linux-gnu)
    "generic",              // CPU
    "+sse4.2",              // Features
    TargetOptions(),
    Optional<Reloc::Model>()
);

return tm;
```

### 6.2.2 Triple Format

The target triple specifies the compilation target:

```
arch-vendor-os-environment

Examples:
x86_64-pc-linux-gnu     # 64-bit Linux on x86
aarch64-apple-darwin    # Apple Silicon macOS
riscv64-unknown-elf     # RISC-V 64-bit bare metal
armv7-unknown-linux-gnueabihf  # ARMv7 Linux with hardware FP
```

### 6.2.3 Cross-Compilation

```bash
# Compile for ARM64
clang --target=aarch64-linux-gnu -o hello_arm hello.c

# Compile for RISC-V
clang --target=riscv64-unknown-elf -o hello_riscv hello.c

# Compile for WebAssembly
clang --target=wasm32 -o hello_wasm hello.c
```

## 6.3 Instruction Selection

Instruction selection maps LLVM IR operations to target machine instructions.

### 6.3.1 DAG-Based Instruction Selection

LLVM uses a SelectionDAG to represent instructions as a directed acyclic graph:

```llvm
; LLVM IR
%result = add i32 %a, %b

; SelectionDAG representation
%a (i32) ──┐
           ├── ADD ──▶ %result (i32)
%b (i32) ──┘

; After instruction selection (x86)
movl  %eax, %ecx    ; %a in eax, result in ecx
addl  %ebx, %ecx    ; %b in ebx, add to ecx
```

### 6.3.2 Pattern Matching

Target instructions are specified using TableGen pattern matching:

```tablegen
// X86InstrInfo.td
def add32rm : I<0x03, MRMDestMem,
  (outs GR32:$dst), (ins i32mem:$src1, GR32:$src2),
  "addl {$src2, $src1|$src1, $src2}",
  [(set GR32:$dst, (add GR32:$src2, (loadi32 addr:$src1)))]>;
```

### 6.3.3 Custom Instruction Selection

For custom frontends, you can use the LLVM CBE (C Backend) or write your own selection:

```cpp
// Simple instruction selection for expressions
class SimpleISel : public SelectionGraphBuilder {
public:
    void selectExpr(SDValue node) {
        switch (node->getOpcode()) {
            case ISD::ADD:
                selectAdd(node);
                break;
            case ISD::SUB:
                selectSub(node);
                break;
            case ISD::MUL:
                selectMul(node);
                break;
            // ... etc
        }
    }
    
private:
    void selectAdd(SDValue node) {
        // Generate target-specific add instruction
        TLI->EmitInstruction(DAG.getTargetMachine(), ...)
    }
};
```

## 6.4 Register Allocation

Register allocation assigns virtual registers to physical CPU registers.

### 6.4.1 Why Register Allocation Matters

| Approach | Pros | Cons |
|----------|------|------|
| **Unallocated (SSA)** | Simple, easy optimization | Can't execute directly |
| **Stack allocation** | Always works | Slow memory access |
| **Register allocation** | Fast execution | Limited registers |

### 6.4.2 Register Classes

Different instruction classes use different register sets:

```tablegen
// Define register classes
def GR32 : RegisterClass<"X86", [i32], 32,
  (add EAX, ECX, EDX, EBX, ESI, EDI, R8D, R9D, R10D, R11D,
       R12D, R13D, R14D, R15D)>;

def FR32 : RegisterClass<"X86", [f32], 32,
  (add XMM0, XMM1, XMM2, XMM3, XMM4, XMM5, XMM6, XMM7)>;
```

### 6.4.3 Allocation Algorithms

**1. Linear Scan (Simple)**

```
Virtual Registers: v1, v2, v3, v4
Physical Registers: r1, r2, r3, r4

Algorithm:
  1. Sort by lifetime start
  2. Allocate sequentially
  3. Spill if no free register
```

**2. Graph Coloring (Optimal)**

```
  v1 ─── v2        Build interference graph
  │╲    ╱│        (edges for overlapping lifetimes)
  v3───v4

  Try to color with 4 colors
  If impossible, spill some vertex
```

**3. LLVM's Default Allocator (Local + Global)**

```bash
# Use specific allocator
llc -regalloc=fast input.ll    # Fast, simple
llc -regalloc=greedy input.ll  # Better quality (default)
llc -regalloc=pbess input.ll   # Optimal (slow)
```

### 6.4.4 Spilling

When physical registers are exhausted, values are "spilled" to memory:

```assembly
; Spilling v1 to stack
; Original: add %v1, %v2
; Spilled:
movl  -4(%rbp), %eax    ; Load v1 from stack
addl  %v2, %eax          ; Add v2
movl  %eax, -4(%rbp)     ; Store result (spill)
```

## 6.5 Calling Conventions

Calling conventions define how functions call each other.

### 6.5.1 Common Calling Conventions

| Convention | Parameter Order | Stack Cleanup | Used By |
|-----------|----------------|---------------|---------|
| **System V (x86-64)** | rdi, rsi, rdx, rcx, r8, r9, stack | Callee | Linux, macOS |
| **Microsoft x64** | rcx, rdx, r8, r9, stack | Caller | Windows |
| **FastCall** | ecx, edx, stack | Caller | Legacy |
| **ARM AAPCS** | r0-r3, stack | Callee | ARM EABI |
| **RISC-V EABI** | a0-a7, stack | Callee | RISC-V |

### 6.5.2 System V x86-64 Calling Convention

```c
// Caller responsibilities
void caller() {
    // 1. Save caller-saved registers if needed
    // 2. Pass arguments in registers (or stack)
    //    %rdi = arg1
    //    %rsi = arg2
    //    %rdx = arg3
    //    %rcx = arg4
    //    %r8  = arg5
    //    %r9  = arg6
    //    More args on stack
    // 3. Call function
    // 4. Receive return value in %rax
    // 5. Clean up stack (if required)
}
```

### 6.5.3 Function Prologue and Epilogue

```assembly
# x86-64 function prologue
pushq  %rbp              ; Save base pointer
movq   %rsp, %rbp        ; Set up new frame
subq   $32, %rsp         ; Allocate local space

# x86-64 function epilogue
addq   $32, %rsp         ; Deallocate locals
popq   %rbp              ; Restore base pointer
ret                        ; Return to caller
```

### 6.5.4 Implementing Calling Conventions in IR

```llvm
; Specify calling convention
define i32 @function(i32 %a, i32 %b) #0 {
  ; Use fastcc for faster, non-standard calling
  ; convention (no stack-based args)
}

attributes #0 = { "noinline" "frame-pointer"="none" }
```

## 6.6 Stack Frame Layout

### 6.6.1 Stack Frame Structure (x86-64)

```
High addresses
┌─────────────────────┐
│   Function Return   │
├─────────────────────┤
│   Previous RBP      │
├─────────────────────┤
│   Local Variables   │
├─────────────────────┤
│   Saved Registers   │
├─────────────────────┤
│   Outgoing Args     │  ← rsp (after allocation)
└─────────────────────┘
Low addresses
```

### 6.6.2 Stack Allocation in LLVM IR

```llvm
; Allocate on stack (alloca)
define i32 @example() {
entry:
  %x = alloca i32           ; Allocate one i32
  %arr = alloca i32, i32 10 ; Allocate array of 10
  
  store i32 42, i32* %x     ; Store to stack
  %val = load i32, i32* %x  ; Load from stack
  
  ret i32 %val
}
```

### 6.6.3 Frame Index

LLVM uses frame indices to refer to stack-allocated variables:

```llvm
; LLVM IR with frame indices
define i32 @example() {
entry:
  %x = alloca i32, align 4           ; FI: 0
  %y = alloca i32, align 4           ; FI: 1
  store i32 10, i32* %x
  %a = load i32, i32* %x
  store i32 %a, i32* %y
  %b = load i32, i32* %y
  ret i32 %b
}
```

## 6.7 Code Generation with LLVM

### 6.7.1 Using the CodeGen APIs

```cpp
#include <llvm/Target/TargetMachine.h>
#include <llvm/Target/TargetOptions.h>
#include <llvm/CodeGen/CommandFlags.h>
#include <llvm/CodeGen/TargetPassConfig.h>
#include <llvm/CodeGen/MachineModuleInfo.h>

class CodeGenBackend {
public:
    bool compileToFile(Module* module, const std::string& outputPath) {
        // Get target machine
        Triple triple(module->getTargetTriple());
        std::string error;
        const Target* target = TargetRegistry::lookupTarget("", triple, error);
        
        if (!target) {
            errs() << "Error: " << error << "\n";
            return false;
        }
        
        TargetOptions options;
        options.OptimizeForSize = false;
        options.MCOptions.ShowMCEncoding = false;
        options.MCOptions.AsmVerbose = true;
        
        auto tm = target->createTargetMachine(
            triple,
            "generic",      // CPU
            "",             // Features
            options,
            Optional<Reloc::Model>()
        );
        
        module->setDataLayout(tm->createDataLayout());
        module->setTargetTriple(tm->getTargetTriple().getTriple());
        
        // Emit assembly or object code
        legacy::PassManager passManager;
        
        if (outputPath.ends_with(".s")) {
            // Emit assembly
            passManager.add(createTargetTransformInfoWrapperPass(
                tm->getTargetIRAnalysis()));
            passManager.add(new TargetPassConfig(*tm, *tm->getTargetLowering()));
            passManager.add(createFileWriter(std::string(outputPath)));
        }
        
        return passManager.run(*module);
    }
};
```

### 6.7.2 Complete Compilation Pipeline

```cpp
// complete_compiler.cpp
#include <iostream>
#include <fstream>
#include "Lexer.h"
#include "Parser.h"
#include "SemanticAnalyzer.h"
#include "LLVMIRGenerator.h"

using namespace llvm;

bool compileFile(const std::string& inputPath, const std::string& outputPath,
                 const std::string& targetTriple) {
    // Read input file
    std::ifstream file(inputPath);
    std::string source((std::istreambuf_iterator<char>(file)),
                        std::istreambuf_iterator<char>());
    
    // 1. Lexical Analysis
    std::cout << "Lexing...\n";
    Lexer lexer(source);
    auto tokens = lexer.tokenize();
    
    // 2. Parsing
    std::cout << "Parsing...\n";
    Parser parser(tokens);
    auto ast = parser.parse();
    
    // 3. Semantic Analysis
    std::cout << "Analyzing...\n";
    SemanticAnalyzer analyzer;
    analyzer.analyze(ast);
    
    // 4. Generate LLVM IR
    std::cout << "Generating IR...\n";
    LLVMIRGenerator irGen;
    auto module = irGen.generate(ast);
    module->setTargetTriple(targetTriple);
    
    // 5. Create Target Machine
    std::cout << "Creating target machine...\n";
    Triple triple(targetTriple);
    std::string error;
    const Target* target = TargetRegistry::lookupTarget("", triple, error);
    
    if (!target) {
        errs() << "Error: " << error << "\n";
        return false;
    }
    
    TargetMachine* tm = target->createTargetMachine(
        triple, "generic", "", TargetOptions(), Optional<Reloc::Model>());
    
    // 6. Emit assembly
    std::cout << "Emitting assembly...\n";
    legacy::PassManager passManager;
    
    FileModel fileModel;
    if (target->addPassesToEmitFile(passManager, fileModel, nullptr,
                                    TargetMachine::CGFT_AssemblyFile)) {
        errs() << "Cannot emit file type\n";
        return false;
    }
    
    fileModel.Output = openOutputFile(outputPath);
    passManager.run(*module);
    
    std::cout << "Compilation successful: " << outputPath << "\n";
    return true;
}

int main(int argc, char** argv) {
    if (argc < 3) {
        std::cerr << "Usage: " << argv[0] 
                  << " <input> <output> [target-triple]\n";
        return 1;
    }
    
    std::string targetTriple = argc > 3 ? argv[3] : 
        Triple::getDefaultTriple().getTriple();
    
    return compileFile(argv[1], argv[2], targetTriple) ? 0 : 1;
}
```

## 6.8 Assembly Output Analysis

### 6.8.1 x86-64 Assembly Structure

```assembly
    .file    "test.c"
    .text
    .globl   main
    .type    main, @function

main:
.LFB0:                     # Function begin
    .cfi_startproc          # Call frame info
    pushq   %rbp            # Save frame pointer
    .cfi_def_cfa_offset 16
    movq    %rsp, %rbp       # Setup new frame
    .cfi_def_cfa_register 6
    
    subq    $16, %rsp        # Allocate locals
    .cfi_offset 6, -16
    
    movl    $5, -4(%rbp)     # int a = 5;
    movl    $3, -8(%rbp)     # int b = 3;
    movl    -8(%rbp), %eax   # Load b
    addl    -4(%rbp), %eax   # Add a
    movl    %eax, -12(%rbp)  # Store result
    
    movl    $0, %eax         # return 0;
    addq    $16, %rsp         # Deallocate
    popq    %rbp              # Restore frame
    .cfi_def_cfa 7, 8
    ret                      # Return
    .cfi_endproc

.LFE0:                     # Function end
    .size    main, .-main
```

### 6.8.2 Understanding Assembly Output

| Directive | Purpose |
|-----------|---------|
| `.globl` | Symbol visibility |
| `.type` | Symbol type (@function) |
| `.cfi_*` | Call frame information for debuggers |
| `.LFB*` | Local function begin labels |
| `.LFE*` | Local function end labels |

## 6.10 Object File Format

### 6.10.1 ELF Structure

```
ELF Header
┌─────────────────────────────────────┐
│ Magic number (0x7F "ELF")          │
│ Architecture (x86-64, ARM, etc.)    │
│ Entry point                         │
│ Program header table offset         │
│ Section header table offset         │
└─────────────────────────────────────┘

Program Headers (segments for execution)
┌─────────────────────────────────────┐
│ LOAD: segments to load into memory │
│ NOTE: additional metadata           │
└─────────────────────────────────────┘

Sections (for linking)
┌─────────────────────────────────────┐
│ .text: executable code              │
│ .data: initialized data             │
│ .bss: uninitialized data            │
│ .symtab: symbol table               │
│ .strtab: string table               │
│ .debug_*: debug information         │
└─────────────────────────────────────┘

Section Headers
```

### 6.10.2 Symbol Table

| Symbol Type | Description |
|-------------|-------------|
| **STT_NOTYPE** | No type (absolute symbols) |
| **STT_OBJECT** | Data object (variables) |
| **STT_FUNC** | Function |
| **STT_SECTION** | Section symbol |
| **STT_FILE** | File name |

### 6.10.3 Linking

```bash
# Link multiple object files

ld main.o -lm -lpthread -o program

# Create shared library
ld -shared -o libtest.so test.o

# Static linking

```

## 7.2 Compiler Design Patterns

### 7.2.1 Visitor Pattern for AST Traversal

The Visitor pattern allows separating algorithms from object structure:

```cpp
// Visitor interface
class ASTVisitor {
public:
    virtual void visit(IntegerLiteral* node) = 0;
    virtual void visit(BinaryExpr* node) = 0;
    virtual void visit(CallExpr* node) = 0;
    // ... other visit methods
};

// Accept method in AST nodes
class Expr {
public:
    virtual void accept(ASTVisitor* visitor) = 0;
};

class BinaryExpr : public Expr {
    void accept(ASTVisitor* visitor) override {
        visitor->visit(this);
    }
};
```

### 7.2.2 Builder Pattern for IR Construction

The Builder pattern simplifies complex object construction:

```cpp
class IRBuilder {
public:
    IRBuilder& setInsertPoint(BasicBlock* bb) {
        // Set insertion point
        return *this;
    }
    
    Value* CreateAdd(Value* lhs, Value* rhs) {
        // Create add instruction
        return result;
    }
    
    // Fluent interface
    IRBuilder& CreatePrologue() {
        CreateAlloca(...);
        CreateStore(...);
        return *this;
    }
};
```

### 7.2.3 Pass Pattern for Optimization

The Pass pattern structures compiler transformations:

```cpp
class Pass {
public:
    virtual bool runOnFunction(Function& F) = 0;
};

class DCEPass : public Pass {
public:
    bool runOnFunction(Function& F) override {
        bool changed = false;
        for (auto& BB : F) {
            for (auto& I : BB) {
                if (isDead(&I)) {
                    I.eraseFromParent();
                    changed = true;
                }
            }
        }
        return changed;
    }
};
```

## 7.3 Best Practices

### 7.3.1 Error Handling

1. **Early Detection**: Catch errors as soon as possible
2. **Clear Messages**: Include line numbers and suggestions
3. **Error Recovery**: Continue parsing to find more errors
4. **Graceful Degradation**: Generate partial output when possible

```cpp
void error(const std::string& msg, int line, int col) {
    std::cerr << "Error at " << line << ":" << col << ": " << msg << "\n";
    errorCount++;
}

void warning(const std::string& msg, int line, int col) {
    std::cerr << "Warning at " << line << ":" << col << ": " << msg << "\n";
    warningCount++;
}
```

### 7.3.2 Performance Considerations

1. **Lazy Parsing**: Parse on demand when possible
2. **Memory Management**: Reuse nodes and strings
3. **Incremental Compilation**: Reuse analysis results
4. **Parallelization**: Process independent units concurrently

### 7.3.3 Testing Strategies

| Test Type | Purpose | Example |
|-----------|---------|---------|
| **Unit Tests** | Test individual components | Lexer token generation |
| **Integration Tests** | Test component interaction | Full compilation pipeline |
| **Golden Tests** | Compare output to expected | IR generation correctness |
| **Fuzzing** | Random input testing | Crash detection |
| **Performance Tests** | Benchmark compilation speed | Large file compilation |

```cpp
// Example: Unit test for lexer
void test_integer_lexer() {
    Lexer lexer("42");
    auto tokens = lexer.tokenize();
    
    assert(tokens.size() == 2);  // NUMBER + EOF
    assert(tokens[0].type == TokenType::NUMBER);
    assert(tokens[0].value == 42);
}
```


### Common IR Patterns

```llvm
; Function with locals
define i32 @foo(i32 %arg) {
  %local = alloca i32
  store i32 %arg, i32* %local
  %val = load i32, i32* %local
  ret i32 %val
}

; Control flow
if:
  %cmp = icmp slt i32 %a, %b
  br i1 %cmp, label %then, label %else
then:
  ret i32 1
else:
  ret i32 0
```

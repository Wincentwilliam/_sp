## 1.1 What Is a Compiler?

A **compiler** is a specialized program that translates source code written in a high-level programming language into a form that can be executed by a computer. This translation process involves multiple sophisticated stages, each transforming the program in a specific way.

### The Compilation Pipeline

Modern compilers are not monolithic single-stage programs. They are organized as a pipeline of interconnected phases:

```
┌─────────────────────────────────────────────────────────────────┐
│                      SOURCE CODE                                │
│                         (.c, .cpp)                              │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                   LEXICAL ANALYSIS                              │
│              (Scanner / Lexer)                                  │
│           Tokens: if, identifier, (, {, etc.                    │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                  SYNTAX ANALYSIS                                │
│                  (Parser)                                       │
│         Abstract Syntax Tree (AST)                              │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                 SEMANTIC ANALYSIS                              │
│            Type Checking, Scope Analysis                        │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│              INTERMEDIATE REPRESENTATION                        │
│                     (LLVM IR)                                   │
│              Platform-independent code                          │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                    OPTIMIZATION                                 │
│            (Various optimization passes)                        │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│               TARGET CODE GENERATION                            │
│                    (Assembly/Object)                            │
│                x86, ARM, RISC-V, etc.                           │
└─────────────────────────┬───────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                      EXECUTABLE                                 │
│                    (Machine Code)                               │
└─────────────────────────────────────────────────────────────────┘
```

## 1.2 The Stages of Compilation

### 1.2.1 Lexical Analysis (Scanning)

Lexical analysis is the first phase of compilation. The **scanner** (or lexer) reads the source code as a stream of characters and groups them into meaningful **tokens**.

**Example:**

```c
// Source code
if (x > 0) {
    return x * 2;
}
```

**Tokens produced:**

| Token Type | Value |
|------------|-------|
| KEYWORD | `if` |
| LEFT_PAREN | `(` |
| IDENTIFIER | `x` |
| GREATER_THAN | `>` |
| INTEGER | `0` |
| RIGHT_PAREN | `)` |
| LEFT_BRACE | `{` |
| KEYWORD | `return` |
| IDENTIFIER | `x` |
| MULTIPLY | `*` |
| INTEGER | `2` |
| SEMICOLON | `;` |
| RIGHT_BRACE | `}` |

The scanner must also handle:
- Whitespace and newline characters (typically ignored)
- Comments (removed during scanning)
- Error handling for invalid characters

### 1.2.2 Syntax Analysis (Parsing)

The **parser** takes the stream of tokens and analyzes them according to the grammar rules of the language. The result is typically an **Abstract Syntax Tree (AST)**, a hierarchical tree representation of the program's structure.

**AST for the example code:**

```
        if-statement
           /    |    \
         /      |      \
        ▼       ▼       ▼
   condition  then    (implicit else)
      / \       |
     /   \      ▼
    x     0   return
                |
                ▼
              multiply
              /      \
             ▼        ▼
            x         2
```

### 1.2.3 Semantic Analysis

Semantic analysis ensures that the program makes sense beyond just syntactic structure. This phase includes:

- **Type checking**: Ensuring operations are applied to compatible types
- **Scope analysis**: Resolving variable references
- **Symbol table management**: Tracking declarations
- **Constant folding**: Computing constant expressions at compile time

```c
// Type checking example
int x = 5;
float y = 3.14;
x = x + y;  // Semantic error: implicit conversion or type mismatch
```

### 1.2.4 Intermediate Representation (IR)

Modern compilers use an **Intermediate Representation (IR)** that is:
- Independent of both the source language and target machine
- Well-suited for optimization
- Easier to analyze and transform than raw source or assembly

LLVM uses its own IR called **LLVM IR** (discussed in detail in Chapter 5).

### 1.2.5 Optimization

Optimization passes transform the IR to:
- Remove redundant computations
- Simplify expressions
- Inline function calls
- Vectorize loops
- And many other transformations

### 1.2.6 Code Generation

The final phase translates the optimized IR to **target-specific assembly code** for the intended architecture (x86, ARM, RISC-V, etc.).

## 1.3 Compiler Architecture Patterns

### 1.3.1 Three-Pass Compiler

Traditional compilers often follow a three-pass structure:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Frontend   │────▶│   Optimizer  │────▶│   Backend    │
│ (Language)  │     │   (IR)       │     │  (Machine)   │
└──────────────┘     └──────────────┘     └──────────────┘
```

| Pass | Responsibility |
|------|----------------|
| **Frontend** | Analysis: Source → AST → IR (language-specific) |
| **Optimizer** | Transformation: IR → Optimized IR (language-independent) |
| **Backend** | Synthesis: IR → Assembly (target-specific) |

This separation allows a single optimizer and backend to work with multiple frontends (e.g., C, C++, Rust all using LLVM).

### 1.3.2 LLVM's Architecture

LLVM exemplifies this modular design:

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   Clang     │  │   Swift     │  │   Rust      │
│   (C/C++)   │  │   Compiler  │  │   Compiler  │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       ▼                ▼                ▼
┌─────────────────────────────────────────────────┐
│              LLVM Core Libraries                │
│  ┌─────────────┐  ┌─────────────┐  ┌──────────┐  │
│  │     IR      │  │ Optimizer  │  │  Target  │  │
│  │ (LLVM IR)   │  │   Passes    │  │ Infos   │  │
│  └─────────────┘  └─────────────┘  └──────────┘  │
└─────────────────────────────────────────────────┘
       │                │                │
       ▼                ▼                ▼
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   x86       │  │    ARM      │  │   RISC-V   │
└─────────────┘  └─────────────┘  └─────────────┘
```

## 1.4 A Simple Compiler Example

Let's build intuition with a minimal compiler that evaluates simple arithmetic expressions. This will be expanded throughout subsequent chapters.

### 1.4.1 Simple Expression Language

We define a tiny language for demonstration:

```
expression := number
           |  expression '+' expression
           |  expression '*' expression
           |  '(' expression ')'
```

### 1.4.2 Implementation in C++

```cpp
#include <iostream>
#include <string>
#include <cctype>

// Simple lexer for our expression language
class Lexer {
public:
    Lexer(const std::string& input) : input(input), pos(0) {}
    
    enum class TokenType { NUMBER, PLUS, STAR, LPAREN, RPAREN, END };
    
    struct Token {
        TokenType type;
        double value;
    };
    
    Token nextToken() {
        skipWhitespace();
        
        if (pos >= input.size()) {
            return {TokenType::END, 0};
        }
        
        char c = input[pos];
        
        if (std::isdigit(c)) {
            return {TokenType::NUMBER, parseNumber()};
        }
        
        pos++;  // Consume character
        
        switch (c) {
            case '+': return {TokenType::PLUS, 0};
            case '*': return {TokenType::STAR, 0};
            case '(': return {TokenType::LPAREN, 0};
            case ')': return {TokenType::RPAREN, 0};
            default:
                std::cerr << "Unknown character: " << c << std::endl;
                return {TokenType::END, 0};
        }
    }

private:
    std::string input;
    size_t pos;
    
    void skipWhitespace() {
        while (pos < input.size() && std::isspace(input[pos])) {
            pos++;
        }
    }
    
    double parseNumber() {
        double result = 0;
        while (pos < input.size() && std::isdigit(input[pos])) {
            result = result * 10 + (input[pos] - '0');
            pos++;
        }
        return result;
    }
};

// Recursive descent parser
class Parser {
public:
    Parser(Lexer& lexer) : lexer(lexer) {
        currentToken = lexer.nextToken();
    }
    
    double parseExpression() {
        return parseAddSub();
    }
    
private:
    Lexer& lexer;
    Lexer::Token currentToken;
    
    void advance() {
        currentToken = lexer.nextToken();
    }
    
    double parseAddSub() {
        double left = parseMulDiv();
        
        while (currentToken.type == Lexer::TokenType::PLUS) {
            advance();
            double right = parseMulDiv();
            left = left + right;
        }
        
        return left;
    }
    
    double parseMulDiv() {
        double left = parsePrimary();
        
        while (currentToken.type == Lexer::TokenType::STAR) {
            advance();
            double right = parsePrimary();
            left = left * right;
        }
        
        return left;
    }
    
    double parsePrimary() {
        if (currentToken.type == Lexer::TokenType::NUMBER) {
            double value = currentToken.value;
            advance();
            return value;
        }
        
        if (currentToken.type == Lexer::TokenType::LPAREN) {
            advance();
            double value = parseExpression();
            if (currentToken.type == Lexer::TokenType::RPAREN) {
                advance();
            }
            return value;
        }
        
        return 0;
    }
};

int main() {
    std::string input = "2 + 3 * 4";
    Lexer lexer(input);
    Parser parser(lexer);
    
    double result = parser.parseExpression();
    std::cout << "Result: " << result << std::endl;  // Output: 14
    
    return 0;
}
```

**Compile and run:**

```bash
g++ -std=c++17 simple_compiler.cpp -o simple_compiler
./simple_compiler
# Output: Result: 14
```

## 1.5 Formal Language Basics

### 1.5.1 Regular Expressions and Tokens

Tokens are typically specified using **regular expressions**. LLVM's lexer generator `lex` (or `flex`) uses this approach.

**Examples:**

| Token | Regular Expression |
|-------|-------------------|
| Integer | `[0-9]+` |
| Identifier | `[a-zA-Z_][a-zA-Z0-9_]*` |
| Keyword | `if\|else\|while\|return` |

### 1.5.2 Context-Free Grammars

The syntax of programming languages is described by **Context-Free Grammars (CFGs)**. A CFG consists of:

- **Terminals**: Token types (e.g., `if`, `+`, `identifier`)
- **Non-terminals**: Syntactic constructs (e.g., `expression`, `statement`)
- **Production rules**: How non-terminals expand
- **Start symbol**: The entry point (typically `program`)

**Example grammar for arithmetic:**

```ebnf
expression ::= expression '+' term | term
term       ::= term '*' factor | factor
factor     ::= number | '(' expression ')'
```

### 1.5.3 Ambiguity and Parsing

Grammars can be **ambiguous**, meaning a string may have multiple parse trees. This is problematic for compilers. The dangling else problem is a classic example:

```c
if (a)
    if (b)
        x = 1;
    else
        x = 2;  // Which if does this else belong to?
```

Compilers resolve ambiguities through:
- **Grammar disambiguation rules** (e.g., precedence declarations)
- **Parser conventions** (shift-reduce prefers shift)
- **AST construction** that encodes intended meaning

## 1.6 Error Handling in Compilers

Robust error handling is crucial. Compilers should:

1. **Detect errors early**: Frontend errors before optimization
2. **Report clear messages**: Include line numbers, column positions, and suggestions
3. **Continue compilation**: Find multiple errors in one run (when possible)
4. **Recover gracefully**: Maintain parser state after errors

```cpp
// Example error message
error: type mismatch at line 15, column 23
  15 |     int x = y + "hello";
                         ^~~~~
                         |
                    cannot add 'const char*' to 'int'
```

## 1.7 Interpreter vs. Compiler

| Aspect | Interpreter | Compiler |
|--------|-------------|----------|
| **Output** | Executes directly | Produces executable |
| **Execution speed** | Slower (runtime overhead) | Faster |
| **Development speed** | Faster (no compile step) | Slower |
| **Error detection** | Runtime errors | Compile-time errors |
| **Examples** | Python, Ruby, JavaScript | C, C++, Rust, Go |

Note: The distinction is blurry in modern systems:
- **JIT compilers** (Java JVM, .NET CLR) compile at runtime
- **VM-based systems** (Java, C#) compile to bytecode first
- **Transpilers** compile between high-level languages

## Summary

This chapter introduced the fundamental concepts of compiler construction:

1. **Compilation pipeline**: Lexical analysis → Syntax analysis → Semantic analysis → IR generation → Optimization → Code generation

2. **Modular architecture**: Modern compilers separate frontend, optimizer, and backend for flexibility

3. **Key abstractions**: Tokens, ASTs, IR, and symbol tables are essential data structures

4. **Language theory foundations**: Regular expressions for lexing, context-free grammars for parsing

5. **LLVM's role**: Provides production-quality implementations of all compiler phases

The subsequent chapters will dive deep into each phase, using LLVM as our implementation framework.
